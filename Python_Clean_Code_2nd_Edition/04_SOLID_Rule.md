## 1. SOLID 원칙의 개요

* SOLID 원칙은 다음과 같은 5가지의 원칙으로 구성되어 있다.

| 원칙                                  | 설명                                                       |
|-------------------------------------|----------------------------------------------------------|
| 단일 책임 원칙 (Single Responsibility)    | 소프트웨어의 컴포넌트 (특히 클래스) 는 **단 하나의 책임을 져야 한다.**              |
| 개방/폐쇄의 원칙 (Open/Closed)             | 클래스 디자인 시, **로직을 캡슐화** 하여 **확장에는 개방, 수정에는 폐쇄** 되게 해야 한다. |
| 리스코프 치환 원칙 (Liskov's Substition)    | 부모 클래스 대신 **하위 클래스 사용 시에도 정상 동작** 해야 한다.                 |
| 인터페이스 분리 원칙 (Interface Segregation) | 인터페이스 (객체가 노출하는 메서드의 집합) 는 **충분히 작아야 한다.**               |
| 의존성 역전 원칙 (Dependency Inversion)    | 코드가 세부 사항, 구체적인 구현보다는 **API 등에 의존** 해야 한다.               |

## 2. 단일 책임 원칙 (Single Responsiblity)

**단일 책임 원칙 (Single Responsibility Principle, SRP)** 은 각 소프트웨어 컴포넌트 (클래스 등) 가 **유일한 책임** 을 져야 한다는 것이다.

* 유일한 책임이라는 것은 **구체적인 일이 1개** 뿐이라는 것을 의미한다.
* 따라서 결과적으로 **해당 코드 변경이 필요한 이유도 단 1개뿐** 이어야 한다.
* 이 원칙은 **응집력을 향상** 시키는 데 도움이 된다.

단일 책임 원칙을 준수하는 좋은 방법은 **모든 메서드를 서로 다른 클래스로 분리** 하는 것이다.

* 예를 들어, 각자의 책임이 있는 클래스를 만들고, 해당 클래스의 인스턴스들과 교류하는 객체를 만들 수 있다.
* 이 경우, **각 클래스가 캡슐화한 메서드들은 나머지 클래스와 독립적** 이다.

## 3. 개방/폐쇄의 원칙 (Open/Close Principle)

**개방/폐쇄의 원칙 (Open/Close Principle)** 은 **클래스 디자인** 시 다음을 준수해야 한다는 것이다.

* 로직을 캡슐화해야 한다.
* 이를 통해 **확장에는 개방** 되어 있고, **수정에는 폐쇄** 되어 있어야 한다.
* 즉, **새로운 요구사항 발생 시, 해당하는 것을 추가하기만 하면 되고 기존 코드를 수정하지 않아도 되는** 것을 말한다.

개방/폐쇄 원칙 요약

| 원칙    | 설명                                                                |
|-------|-------------------------------------------------------------------|
| 개방 원칙 | 이미 존재하는 코드를 변경하는 대신, **코드를 확장하여 새로운 case (이벤트 유형 등) 를 지원** 해야 한다. |
| 폐쇄 원칙 | 새로운 case (이벤트 유형 등) 지원 시, **메서드를 변경하지 않아야** 한다.                   |

### 3-1. 개방/폐쇄의 원칙을 따르지 않는다면?

* 개방/폐쇄의 원칙을 따르지 않는 경우 다음과 같은 문제점이 발생한다.
  * 계속 커질 수 있는 로직이 특정 메서드에 집중됨 (여러 가지 일을 하게 됨)
  * 새로운 케이스 (이벤트 유형) 추가 시 **메서드를 수정해야 함**
  * ```if-elif-else``` 는 가독성을 해치는 구조
* 개방/폐쇄의 원칙을 따르지 않는 경우는 대표적인 다음과 같다.
  * 클래스의 추상화가 부족하여 **클래스 간 직접 상호 작용이 많음**

## 4. 리스코프 치환 원칙 (Liskov Substitution)

**리스코프 치환 원칙 (Liskov Substitution Principle)** 은 설계 안정성을 위한, 다음을 핵심 내용으로 하는 원칙이다.

* 클라이언트 (다른 class를 호출하는 class) 가 부모 type (class) 사용 가능 시, **부모 type 대신 어떤 자식 type을 사용해도 정상 작동** 해야 한다.
* 클라이언트는 **클라이언트가 호출하는 class의 계층 구조 변경에 완전히 독립적** 이어야 한다.

리스코프 치환 원칙의 위배 사례는 다음과 같다.

* 클라이언트에서 **클래스를 호환되지 않는 방식으로 사용 (파라미터 타입 임의 변경)** 하는 경우

```python
>>> class LanguageModel:
	def __init__(self, param_count: int):
		self.param_count = param_count
	def get_size(self, unit: str):
		if unit == 'B':
			return str(self.param_count / 1_000_000_000) + 'B'
		else:
			return str(self.param_count) + ' params'

		
>>> class VLM(LanguageModel):
	def get_size(self, unit: int):
		return self.param_count / unit
```

* 파라미터 타입이 아닌 **메서드의 서명 (매개변수 목록 등) 자체가 다른 경우**

### 4-1. 부모 클래스와 하위 클래스 간의 계약

하위 클래스는 **부모 클래스가 정의한 클라이언트와의 계약을 준수** 해야 한다. 즉 다음과 같다.

* 하위 클래스의 **사전 조건은 부모 클래스보다 엄격해서는 안 된다.**
* 하위 클래스의 **사후 조건은 최소 부모 클래스 수준** 이어야 한다.
* 즉, 하위 클래스는 부모 클래스에 비해 **사전 조건 (코드 실행 이전 처리) 은 약하고, 사후 조건 (반환값 유효성 검사 등) 은 강하게** 하는 것으로 이해할 수 있다.

## 5. 인터페이스 분리 원칙 (Interface Segregation)

**인터페이스 분리 원칙 (Interface Segregation Principle, ISP)** 은 **인터페이스는 충분히 작아야 한다** 는 원칙이다.

* **인터페이스 (Interface)** 는 객체가 노출하는 메서드의 집합 (객체가 수신/해석 가능한 모든 메시지로 구성됨) 이다.
* 클래스 관련 용어

| 용어                              | 설명                           |
|---------------------------------|------------------------------|
| 추상 기본 클래스 (abstract base class) | 실질적인 무엇인가를 의미하지 않는, 추상화된 클래스 |
| 가상 서브클래스 (virtual subclass)     | 기본 클래스로부터 추가된 파생 클래스         |

* 인터페이스 분리 원칙의 구체적 의미
  * 여러 개의 메서드가 있는 인터페이스는, **메서드 개수가 더 적은 여러 개의 인터페이스로 분할** 하는 것이 좋음
  * 이때, 메서드 개수는 **1개까지 줄이는** 것이 가장 좋음

## 6. 의존성 역전 원칙 (Dependency Inversion)

**의존성 역전 원칙 (DIP)** 은 다음을 의미한다.

* 코드는 세부 사항이나 구현 대신 **API 등에 적응** 해야 한다.
* 세부 사항에 의존하지 않는다. (추상화를 통해)
* 세부 사항은 추상화에 의존하도록 한다.

의존성에서 추상화는 **인터페이스를 통해 제공** 되어야 하며, **구체 클래스 간 상호작용보다는 클래스-인터페이스 간 상호작용을 하도록** 해야 한다.

### 6-1. 의존성 주입

**의존성 주입 (Dependency Injection)** 이란, **클래스가 필요로 하는 것들을 직접 관리하는 대신, 제공받도록 하는 것** 을 말한다.

* 즉, 클래스가 필요로 하는 것을 제공하는 **다른 클래스의 객체를 생성하는 대신, 외부 객체로부터 생성된 정보를 제공받도록** 하는 것이다.

의존성 주입을 하는 이유는 다음과 같다.

* 설계 유연성 향상
* 테스트 용이성 향상 (단위 테스트 등)
