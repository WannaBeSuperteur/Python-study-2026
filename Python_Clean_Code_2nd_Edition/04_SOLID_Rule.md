## 1. SOLID 원칙의 개요

* SOLID 원칙은 다음과 같은 5가지의 원칙으로 구성되어 있다.

| 원칙                                  | 설명                                                       |
|-------------------------------------|----------------------------------------------------------|
| 단일 책임 원칙 (Single Responsibility)    | 소프트웨어의 컴포넌트 (특히 클래스) 는 **단 하나의 책임을 져야 한다.**              |
| 개방/폐쇄의 원칙 (Open/Closed)             | 클래스 디자인 시, **로직을 캡슐화** 하여 **확장에는 개방, 수정에는 폐쇄** 되게 해야 한다. |
| 리스코프 치환 원칙 (Liskov's Substition)    | 부모 클래스 대신 **하위 클래스 사용 시에도 정상 동작** 해야 한다.                 |
| 인터페이스 분리 원칙 (Interface Segregation) | 인터페이스 (객체가 노출하는 메서드의 집합) 는 **충분히 작아야 한다.**               |
| 의존성 역전 원칙 (Dependency Inversion)    | 코드가 세부 사항, 구체적인 구현보다는 **API 등에 의존** 해야 한다.               |

## 2. 단일 책임 원칙 (Single Responsiblity)

**단일 책임 원칙 (Single Responsibility Principle, SRP)** 은 각 소프트웨어 컴포넌트 (클래스 등) 가 **유일한 책임** 을 져야 한다는 것이다.

* 유일한 책임이라는 것은 **구체적인 일이 1개** 뿐이라는 것을 의미한다.
* 따라서 결과적으로 **해당 코드 변경이 필요한 이유도 단 1개뿐** 이어야 한다.
* 이 원칙은 **응집력을 향상** 시키는 데 도움이 된다.

단일 책임 원칙을 준수하는 좋은 방법은 **모든 메서드를 서로 다른 클래스로 분리** 하는 것이다.

* 예를 들어, 각자의 책임이 있는 클래스를 만들고, 해당 클래스의 인스턴스들과 교류하는 객체를 만들 수 있다.
* 이 경우, **각 클래스가 캡슐화한 메서드들은 나머지 클래스와 독립적** 이다.

## 3. 개방/폐쇄의 원칙 (Open/Close Principle)

**개방/폐쇄의 원칙 (Open/Close Principle)** 은 **클래스 디자인** 시 다음을 준수해야 한다는 것이다.

* 로직을 캡슐화해야 한다.
* 이를 통해 **확장에는 개방** 되어 있고, **수정에는 폐쇄** 되어 있어야 한다.
* 즉, **새로운 요구사항 발생 시, 해당하는 것을 추가하기만 하면 되고 기존 코드를 수정하지 않아도 되는** 것을 말한다.
