## 1. 계약에 의한 디자인

**계약에 의한 디자인 (Design by Contract)** 은 다음과 같은 것을 의미한다.

* 소프트웨어 컴포넌트 간 **규칙을 명문화**
* 이 규칙을 위반했을 경우, **계속할 수 없는 명시적 이유를 나타내는 예외** 발생

이때 계약의 구성 요소는 다음과 같다.

| 구성 요소                                  | 설명                                                            |
|----------------------------------------|---------------------------------------------------------------|
| 사전 조건 (pre-condition) **(코드 레벨 강제)**   | 코드 실행 **이전에 처리되어야 하는 조건** (데이터 유효성 검사 등)<br>- 런타임 중에 확인할 수 있음 |
| 사후 조건 (post-condition)  **(코드 레벨 강제)** | 함수 반환값에 대한 유효성 검사                                             |
| 불변식 (invariant)                        | 함수 실행 중 **일정하게 유지** 되어야 하는 것                                  |
| 부작용 (side-effect)                      | 코드의 부작용                                                       |

* 이 계약을 명시하는 경우, **책임 소재 파악이 용이 (예: 사전 조건 위반 시 클라이언트의 책임)** 하다.

## 2. 방어적 프로그래밍

**방어적 프로그래밍 (defensive programming)** 은 코드의 모든 부분 (객체, 함수 등) 을 **유효하지 않은 것으로부터 스스로 보호** 하는 것이다.

* 방어적 프로그래밍의 종류는 다음과 같다.

| 종류                                    | 요약        | 설명                                                                   |
|---------------------------------------|-----------|----------------------------------------------------------------------|
| 예상 가능한 시나리오의 오류 처리                    | 에러 핸들링    | - 값 대체, 오류 로깅, 예외 처리 등<br>- 값 대체의 결과 **일부 오류를 숨길 수 있는데, 이에 주의** 해야 함 |
| 발생하지 않아야 하는 오류 처리 (불가피한 조건에 의해서 발생하는) | assertion | - **항상 참이어야 하는** (거짓인 경우 오류가 되는) bool 조건                             |

* 참고
  * 에러 핸들링 시, **서로 다른 오류를 서로 다른 부분에서 핸들링** 하게 하여 **책임을 분산** 시키는 것이 좋다.
  * 보안을 위해, **API 등의 end user에게 traceback을 노출해서는 절대 안 된다.**
  * **비어 있는 except 블록** 은 어떠한 예외도 그냥 삼키고, 런타임 오류가 발생하지 않으며, 예외에 대한 아무 정보도 확인할 수 없으므로 **최악의 코드** 이다.
  * assertion을 **```try-except``` 등을 이용한 제어 메커니즘으로 사용하면 안 된다.**
  * assertion보다는 예외 처리가 보다 일반적인 프로그래밍 방식이다.

## 3. 파급 효과, 응집력, 결합력

* **파급 (ripple) 효과** 는 소프트웨어의 특정 지점의 변화가 소프트웨어 전체로 전파되는 것이다.
  * 유지보수성 향상을 위해, **파급 효과를 최소화 (관심사 분리)** 해야 한다.
  * 소프트웨어의 각 부분은 **관심사 (특정 기능 등)** 에 대해서만 책임을 지게 해야 한다.
* 응집력과 결합력

| 구분             | 설명                                         | 좋은 경우       |
|----------------|--------------------------------------------|-------------|
| 응집력 (cohesion) | 객체는 **목적이 작고 잘 정의** 되어야 하며 **가능한 한 작아야** 함 | **높을수록** 좋음 |
| 결합력 (coupling) | 2개 이상의 객체 간의 **의존 관계**                     | **낮을수록** 좋음 |

## 4. 개발 지침 약어 및 설명

| 약어        | 설명                                        | 추가 설명                                                                                          |
|-----------|-------------------------------------------|------------------------------------------------------------------------------------------------|
| DRY, OAOO | 코드의 정보는 **한 곳에, 한 번만** 정의되어야 함 (중복 코드 금지) | - 코드 중복 시 **오류/버그, 많은 변경 비용, 다중 데이터 소스에 대한 신뢰성 하락** 등 문제점이 많음<br>- **함수 생성** 등을 통해 코드 중복 방지 가능 |
| YAGNI     | 미래의 요구 사항을 예측한 **과잉 엔지니어링** 자제            | 현재 필요한 기능 관련 코드만 작성                                                                            |
| KIS       | 기능 요구사항에 맞는 **가장 간단한 자료구조** 사용            | Python의 철학: **단순한 것이 복잡한 것보다 낫다**                                                              |
| EAFP      | 일단 실행하고, 발생하는 오류 (동작하지 않음) 에 대응한다         | **보다 Pythonic 한 방법**                                                                           |
| LBYL      | EAFP의 반대로, 실행하기 전에 오류 발생 가능성 등을 살펴본다      |                                                                                                |
